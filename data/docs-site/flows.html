A. Player adds (enters) a tournament

The flagship flow. The user picks a tournament, taps Enter, and gets debited the entry fee. Idempotent on idempotency_key — double-clicks are a single debit.

1

Tap Enter

Web/Unity sends tournament_enter {slug, idempotency_key}

2

Validate

Status (OPEN/ACTIVE), country, US state (not in WA/AZ/CT/NJ/NY/VA), age (≥18), rate-limit (60/min/user)

3

Idempotency check

If a row exists for (user, slug, idempotency_key) → return existing entry, no second debit

4

Debit + write

Read wallet → debit entry_fee_bc → write tournament_entries row → increment tournament_pot.pot_bc

5

Broadcast + return

Realtime 1001:entry on the tournament:{slug} stream + leaderboard upsert (score=0) + return {ok, entry_id, pot_bc, your_rank}

sequenceDiagram autonumber participant U as User (web/Unity) participant N as Nakama RPC participant S as Storage (entries, pot, wallet) participant L as Leaderboard participant P as Push subscribers U->>N: tournament_enter {slug, idempotency_key} N->>N: validate slug + status (OPEN/ACTIVE) N->>N: check geo (country, US state) N->>N: check age (>= 18) N->>N: check rate limit (60/min/user) N->>S: read tournament_entries by idempotency_key alt already exists S-->>N: existing entry row N-->>U: {ok:true, entry_id, idempotent:true} else new entry N->>S: read bc_wallet N->>S: debit entry_fee_bc N->>S: write tournament_entries row N->>S: increment tournament_pot.pot_bc N->>L: leaderboard upsert (score=0) N->>P: realtime broadcast 1001 "entry" N-->>U: {ok:true, entry_id, pot_bc, your_rank} end

What is enforced

  • Status gate: only OPEN or ACTIVE tournaments accept entries; SETTLING returns tournament_closed.
  • Geo gate: country must be in countries_allowed; US state must not be in ENTRY_BLOCK_US_STATES (WA, AZ, CT, NJ, NY, VA).
  • Age gate: Cognito profile must show a DOB ≥ 18 years ago. Violators get age_gated.
  • Idempotency: (user_id, slug, idempotency_key) is unique; second-call returns the original entry row.
  • Insufficient balance: falls back to AMOE flow (see §C below).

B. Pre-enrollment (free, founder cohort)

Pre-enroll is the launch funnel — free, 1-click, and gives the first 1,000 users a "founder" badge plus a 2x first-win multiplier on payouts.

sequenceDiagram autonumber participant U as User participant N as Nakama participant S as Storage participant Q as Quest engine U->>N: tournament_pre_enroll {slug, ref_code?} N->>S: read tournament_pre_enroll(user_id, slug) alt already pre-enrolled S-->>N: existing row N-->>U: {ok:true, idempotent:true, founder:true|false} else first time N->>S: read meta.pre_enroll_count alt count < 1000 N->>S: write tournament_pre_enroll {founder:true} N->>Q: emit "founder_signed_up" quest event else cap hit N->>S: write tournament_pre_enroll {founder:false} end N->>S: meta.pre_enroll_count += 1 opt ref_code present N->>S: write referral_attribution end N-->>U: {ok:true, founder, total_enrolled} end

Pre-enroll is rate-limited at 20/min/user to keep bots from hogging the founder cap. The cron tournament_cron_tick automatically promotes pre-enrolled users into a paid entry the moment a slug flips OPEN, debiting BC if available, otherwise leaving them at amoe_pending.

C. AMOE — Alternative Method Of Entry

Required for US-state-skill-gaming compliance: every paid tournament must offer a free, equivalent path. We use a 6-video Learning Series that gives the user one free entry per tournament. AMOE entrants are flagged in storage but are otherwise indistinguishable from paid entrants on the leaderboard.

sequenceDiagram autonumber participant U as User participant N as Nakama participant V as Learning Series (CDN) participant S as Storage U->>N: learning_track_get {tournament_slug} N-->>U: {videos:[6], progress:{...}} loop 6 videos U->>V: HLS playback U->>N: learning_video_record_watch {video_id, watched_pct} N->>S: write learning_progress end U->>N: learning_check_submit {tournament_slug} N->>S: read learning_progress alt 6/6 complete N->>S: write tournament_entries {paid_via:"amoe", bc_charged:0} N->>S: pot_bc unchanged (AMOE doesn't add to pot) N-->>U: {ok:true, free_entry_granted:true} else < 6 N-->>U: {ok:false, error:"learning_incomplete", missing:[ids]} end

AMOE "no-lose" refund

AMOE entrants who finish in the top-50% (Classic), survive past day-3 (Elimination), or hit ≥3/5 (Pick-N) get their BC entry refunded by the tournament_settle cron. Paid entrants get the standard payout split. This is the legal pillar that keeps us out of the gambling bucket in skill-game-prohibited US states.

D. Pack submission — the play loop

Once entered, a user fetches a quiz pack, plays it, and submits the result. Anti-cheat runs server-side; suspicious submissions are flagged soft_dq and reviewed before payout.

sequenceDiagram autonumber participant U as User participant N as Nakama participant CF as Content Factory participant S as Storage + Leaderboard U->>N: tournament_content_get_pack {slug, pack_index} N->>CF: GET /quiz_pack?id=... CF-->>N: {questions, correct_answers (server-side)} N-->>U: {questions, pack_id, time_limit_sec} Note over U: User plays the pack U->>N: tournament_submit_pack_result {slug, pack_id, answers, client_latency_ms} N->>N: anti-cheat (latency floor 300ms, daily ceiling 200) N->>S: persist tournament_submits N->>S: leaderboard upsert (score = sum of correct + speed bonus) N->>S: write tournament_pot.last_action_at N-->>U: {ok:true, score, your_rank, soft_dq?}

For Pick-N, the equivalent endpoint is tournament_submit_picks. The pick window is enforced server-side (max_pick_window_hours before end_iso); after lock, all submissions return pick_window_closed.

E. Settlement — paying out the pot

At end_iso the cron flips status to SETTLING, computes the final ranking from the leaderboard, applies the per-format split, and writes payout rows. Target runtime is < 60s per tournament. Ledger-drift target is 0.

1

Status flip

tournament_cron_tick sees now ≥ end_iso → meta.status = SETTLING + push 1004:settling

2

Snapshot ranking

Read leaderboard, freeze the order. soft_dq entrants are excluded pending review

3

Compute rake

rake_bc = pot_bc * 0.15 credited to house ledger. Net payout pool = pot_bc - rake_bc

4

Top-N split (Classic)

40/20/10/5/5/5/5/5/2.5/2.5 across top 10 (or equal split for Elimination survivors, or multiplier × entry for Pick-N graders)

5

AMOE refunds

For AMOE entrants who hit the no-lose threshold (top-50% / survive-N / ≥3-of-5), refund their entry_fee_bc

6

Invariant check

Assert sum(payouts) + rake_bc == pot_bc. Any drift triggers a P0 alert and the payout is rolled back

7

Mark settled

meta.status = SETTLED + push 1004:settled with payouts list + send "Your payout: X BC" notification to every winner

sequenceDiagram autonumber participant C as tournament_settle cron participant S as Storage participant L as Leaderboard participant W as bc_wallet participant P as Push C->>S: read meta + entries S-->>C: {pot_bc, format, split_config} C->>L: snapshot final ranking L-->>C: ordered [user_id, score] Note over C: rake = pot * 0.15
net = pot - rake loop top-N split C->>W: credit user_i (share_i x net) C->>S: write payout row {tournament, user, bc, rank} end loop AMOE no-lose alt user qualifies (top-50% / survive-N / ≥3/5) C->>W: credit refund (entry_fee_bc) C->>S: write payout row {paid_via:"amoe_refund"} end end C->>S: meta.status = SETTLED C->>P: realtime broadcast 1004 "settled" C->>P: push "Your payout: X BC"

F. Certificates — the social-share artifact

After settlement, the top-10 (Classic), all survivors (Elimination), or anyone with ≥3/5 (Pick-N) can claim a shareable PNG certificate. The cert is rendered by an AI Lambda and uploaded to intelli-verse-x-media/certificates/.

sequenceDiagram autonumber participant U as User participant N as Nakama participant L as Cert Lambda participant S3 U->>N: tournament_claim_certificate {slug} N->>N: verify settled + user qualifies N->>L: invoke {user, slug, rank, score, payout_bc} L->>L: render PNG (1200x630 OG-ratio) L->>S3: PUT certificates/{slug}/{user_id}.png S3-->>L: signed URL L-->>N: {url, cert_id} N->>N: write tournament_certs row N-->>U: {url, share_text, og_meta} Note over U: User shares to FB/X/LinkedIn

G. Referrals — viral growth

Every user has a unique 6-char referral code. When a friend pre-enrolls with that code, the referrer accumulates points. Top-100 referrers at T+1h after public launch get cash via Tremendous (Top-1: $500, Top-2-3: $250, Top-4-10: $100, Top-11-100: $25).

sequenceDiagram autonumber participant F as Friend (new user) participant U as Referrer participant N as Nakama participant S as Storage participant T as Tremendous U->>N: referral_my_code → SHARE_LINK?ref=AB12CD F->>F: clicks link F->>N: referral_pre_enroll_with_code {ref_code:"AB12CD"} N->>N: lookup referrer by code N->>S: write referral_attribution {referee, referrer, slug} N->>S: increment referral_points[referrer] N->>N: emit quest "referral_made" for U Note over N: T+1h after public launch N->>S: read referral_leaderboard top 100 N->>T: bulk Tremendous order (top 100) T-->>N: tracking IDs per user N->>U: push "Top-1 referrer! $500 inbound"

H. Bracket playoff — top-64 from the qualifier

For the biggest weekly tournaments, the top-64 from the open qualifier feed into a Bracket-rendered single-elimination playoff. Each bracket "match" is a 24h pack-submission window; highest score wins, ties broken by submitted-at.

sequenceDiagram autonumber participant N as Nakama participant B as Bracket FastAPI participant U as Top-64 users N->>B: POST /tournaments {tournament_type:"SINGLE_ELIMINATION", max_players:64} B-->>N: {tournament_id} N->>S: cache bracket_id in tournaments_meta N->>B: POST /players/batch {top-64 with seed scores} B-->>N: {ok:true, matches:[round-1 pairings]} loop 6 rounds (64→32→16→8→4→2→1) N->>U: push "Round X started — submit your pack" U->>N: tournament_submit_pack_result {slug, pack_id, ...} Note over N: 24h window per round N->>N: at round end, compute winner of each match N->>B: PUT /matches/{match_id}/result {winner_user_id, scores} B-->>N: {next_round_ready} end N-->>U: tournament_settle distributes the playoff bonus pot

Bracket is reachable internally at bracket.aicart.svc.cluster.local:8400 and publicly at https://bracket.intelli-verse-x.ai (the iframe view). Auth is a service-to-service JWT injected into the Nakama deployment as BRACKET_SERVICE_JWT.

I. Elimination — the daily-cut format

Every midnight UTC, the bottom-50% of active entrants are eliminated. The cron tournament_eliminate_round handles the cut; eliminated users receive a notification + a "you survived day N" badge.

sequenceDiagram autonumber participant C as tournament_eliminate_round participant L as Leaderboard participant S as Storage participant P as Push Note over C: Triggered at each cut_times_utc[i] C->>L: read current ranking (active only) L-->>C: ordered scores C->>C: bottom 50% by score loop eliminated users C->>S: write tournament_eliminations {user, round, eliminated_at} C->>P: push "You survived to day N. Thanks for playing!" end loop survivors C->>P: push "You survived day N. Day N+1 starts now." end C->>S: meta.survivors_count = N/2 Note over C: At final cut → tournament_settle splits pot equally among survivors

J. Pick-N — the slip format

Players submit a single slip of N picks (Wave-1: 5). Score = number of correct picks; multiplier table determines payout.

sequenceDiagram autonumber participant U as User participant N as Nakama participant S as Storage U->>N: tournament_get_pick_n_questions {slug} N-->>U: {questions:[5 questions], pick_lock_iso} U->>N: tournament_submit_picks {slug, picks:[5 answers]} N->>N: validate before pick_lock_iso N->>S: write tournament_picks (idempotent on user) N->>S: leaderboard upsert (score=0 for now) N-->>U: {ok:true, locked_at} Note over N: At end_iso, settle cron runs N->>N: grade picks (e.g. 4/5) N->>N: lookup multiplier (4/5 → 5x) N->>S: credit user (entry_fee_bc x 5) N->>S: write payout

K. Live pot updates — Nakama realtime

Every entry, settlement, and elimination broadcasts a typed message on the tournament:{slug} stream. Web and Unity subscribe so the UI ticks without polling.

sequenceDiagram autonumber participant W as Web client participant N as Nakama (WS) participant S as Storage W->>N: socket.connect() W->>N: joinChat("tournament:slug") Note over N: User-A submits a pack S->>N: realtime broadcast {op:1001, slug, pot_bc, entries_count} N->>W: stream message {op:1001, ...} W->>W: animate pot ticker Note over N: 5s later — settlement S->>N: realtime broadcast {op:1004, slug, payouts} N->>W: stream message {op:1004, ...} W->>W: render winners modal

Op codes: 1001=entry, 1002=submit, 1003=elimination, 1004=settled, 1005=cert_ready. Subscriber TTL is 12h (set in storage.ts) so stale subs from sign-outs don't accumulate.

L. Edge cases & how the system handles them

ScenarioWhat happens
User clicks "Enter" 2× quicklyIdempotency key dedupes — single debit, single entry row.
User in a US-blocked state attempts entryReturns 403 entry_blocked_in_us_state; UI shows the AMOE alternative.
User has 0 BC and tries to enterReturns insufficient_balance; UI offers Buy-BC sheet or AMOE path.
Anti-cheat trips on impossible latencySubmission accepted but flagged soft_dq=true; settle cron excludes from payouts pending review.
Settlement fails mid-payout (pod crash)Idempotent re-run safe; payout rows have unique (slug, user_id, rank) key.
Bracket service unreachable during seedLogged as bracket_unreachable; cron retries every 5 min for up to 1h.
User signs out during a tournamentEntry persists; on re-sign-in they see their pending submissions and rank.
Pre-enroll cap hit (1,001st user)User is enrolled but flagged founder:false; no 2x multiplier.
Tournament ends with 0 entriesHouse recovers seed; settle is a no-op; status flips to ARCHIVED at T+30d.
Daily reset (gk-royale-daily, pick-5-daily)Cron creates a new instance with the same slug + a date suffix; old instance settles independently.
