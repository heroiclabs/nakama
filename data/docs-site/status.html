1. Launch blockers (P0 — fix before Jul 1)

Each item below was verified live. None of these are theoretical — every one is a check that failed.

B1 · Tournament cron jobs are NOT deployed in Kubernetes

probe: kubectl get cronjobs -A | grep tournament → 0 matches · severity: P0

tournament_cron_tick, tournament_settle, tournament_eliminate_round, and tournament_cron_pregen are registered as RPCs inside Nakama, but no Kubernetes CronJob is invoking them. Today there are 28 CronJobs in the aicart namespace; none are tournament-related.

Impact. On July 1, 04:00 UTC, the 11 tournaments will not automatically transition from PRE_ENROLL → OPEN. Daily/weekly settlements won't run. Survivor cuts won't fire. Content pre-generation backlog won't drain.

Fix: ship 4 K8s CronJob manifests in intelli-verse-kube-infra: tick (every minute), settle (every 5 min), eliminate (every minute during ACTIVE windows), pregen (every 10 min). All four can call the existing internal RPCs via curl http://intelliverse-nakama:7350/v2/rpc/... with the runtime HTTP key already in nakama-secret.

B2 · Lambda tournament_certificate is NOT deployed

probe: aws lambda list-functions --region us-east-1 → 4 fns; none with "tournament" in the name · severity: P0

Only four Lambdas exist in us-east-1/970547373533: extract-and-deploy-webgl-game, ivx-push-provider-bridge, send, register. The runbook §3.1 T-24h checklist explicitly requires "Lambda tournament_certificate deployed + smoke-tested."

Impact. Top-10 finishers in every tournament will have no shareable PNG certificate, blocking the viral-share loop that flows.html assumes.

Fix: the source code is staged at nakama/data/lambda-functions/tournament_certificate — deploy via aws lambda create-function + S3 trigger to intelli-verse-x-media/quiz-verse/tournament-certificates/.

B3 · Bracket service JWT expires before launch

probe: env BRACKET_SERVICE_JWT in nakama pod → exp claim = 1780465694 = Jun 4 2026 · severity: P0

The Nakama → Bracket S2S bridge uses BRACKET_SERVICE_JWT mounted via env. Decoded payload: {"user":"admin@intelli-verse-x.ai","exp":1780465694}. That's ~6 days from now, three weeks before public launch on Jul 1.

Impact. When the JWT expires, every Survivor cut, bracket advancement, and elimination flow will start returning 401 from the bracket service. Silent until something tries to advance a round.

Fix: rotate to a token with exp ≥ 2026-08-01, redeploy nakama pods so the new env is picked up, then add a quarterly rotation reminder.

B4 · AMOE Learning Series videos are NOT staged in S3

probe: aws s3 ls s3://intelli-verse-x-media/quiz-verse/ → no learning-series/ prefix; aws s3api list-objects-v2 ... | grep amoe|learning|cert → 0 matches · severity: P0

The AMOE compliance path requires a 6-video Learning Series so users can earn an entry without paying. Today there is no learning-series/ prefix in intelli-verse-x-media/quiz-verse/, and the audit page assumes those videos exist.

Impact. Without an equivalent free path, paid entry from US states with sweepstakes restrictions is legally exposed (Key2Law 2026: "AMOE must be equivalent and accessible").

Fix: generate 6 short videos via content-factory-api's viral_lesson_short endpoint (one per launch-slate topic), upload to s3://intelli-verse-x-media/quiz-verse/learning-series/{slug}/{1..6}.mp4, and verify tournament_amoe_completion RPC accepts them.

B5 · Unity Tournament screens are NOT wired into any scene

probe: grep -l "TournamentScreen\|TournamentManager" Assets/Scenes/*.unity → 0 matches · severity: P0 for mobile launch

Twelve C# scripts exist under Assets/_QuizVerse/Scripts/Tournaments/ (TournamentManager, TournamentScreen, PreEnrollScreen, BracketViewerScreen, LearningSeriesPlayer, etc.) but a grep across Assets/Scenes/*.unity finds zero references. There are also no tournament .prefab assets in the project.

Impact. The Unity Android/iOS client will not show the tournaments lobby. Web is fine; mobile is not.

Fix: add a Screen_TournamentsLobby GameObject under Screen_Canvas/Quizzes/ in MainQuiz.unity, attach TournamentScreen, wire it to the existing screen-stack manager, and bind the prefab navigations.

B6 · 7 authenticated RPCs leak Goja stack traces on anonymous calls

probe: anonymous POST to each RPC → HTTP 500 with "Error: User ID is required at requireUserId (index.js:132420:19)" · severity: P1 (UX) + P2 (info disclosure)

The seven affected RPCs:

tournament_enter, tournament_pre_enroll, tournament_submit_pack_result, tournament_submit_picks, tournament_claim_cert, tournament_leaderboard_around_me, tournament_leaderboard_friends, tournament_leaderboard_tier_league

All raise an uncaught exception via requireUserId(). Nakama surfaces it as HTTP 500 with the full stack trace including internal file path and line numbers — bad UX for anonymous web users and information-disclosure for security review.

Fix: wrap each handler in a try/catch that maps unauthenticated errors to a clean {"code":"unauthenticated","message":"sign in required"} 401 response. One-line patch at the top of requireUserId in data/modules/src/lib/auth.ts (or whichever module owns it) plus a global RPC outer wrapper.

B7 · Daily quiz manifest for today returns 404 in S3

probe: aws s3api head-object --bucket intelli-verse-x-media --key daily-quiz/dailyquiz-2026-05-29.json → 404 Not Found · severity: P0 once tournaments open

The doc site references the daily-quiz manifest at s3://intelli-verse-x-media/daily-quiz/dailyquiz-{yyyy-MM-dd}.json as the source for the daily-tournament question pack. Today's key (2026-05-29) does not exist; only Russian-language Premier-League manifests are present in quiz-verse/daily/.

Impact. If gk-royale-daily opens on Jul 1 with no manifest in S3, the entry flow returns "no quiz pack" and the player can't play.

Fix: add a daily-pregen CronJob (5 min before midnight UTC) that calls tournament_cron_pregen and uploads the pack to the documented S3 key. Rolls up under B1 above.

4. Reproducing the probes

        -o jsonpath='{.items[0].metadata.name}')
HTTP_KEY=$(kubectl get secret -n aicart nakama-secret \
        -o jsonpath='{.data.http_key}' | base64 -d)
